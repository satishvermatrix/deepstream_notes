DeepStream VSS Pipeline Architecture

DeepStream VSS Pipeline Architecture

Production-grade architecture for multi-camera video analytics with Detection, Tracking, ReID, License Plate OCR, and top-frame/crop extraction — all metadata flowing to Kafka.

Design Principles

  • DeepStream owns what only DeepStream can do: GPU inference, frame-synchronized tracking, raw pixel access
  • One probe, fire-and-forget: the pipeline thread never blocks on I/O or encoding
  • Thin pipeline, smart consumers: detection and tracking in DeepStream; ReID, OCR, aggregation downstream
  • Crops never go through nvmsgconv: binary/image data takes the custom producer path
  • Bounded queues with drop policy: lose crops before losing frames

Probe Implementation

One probe on the sink pad of the secondary detector. Returns immediately — no blocking I/O.

def unified_probe(pad, info, u_data):
    batch_meta = pyds.gst_buffer_get_nvds_batch_meta(info.get_buffer())

    for frame_meta in nvds_frame_meta_list(batch_meta):
        for obj_meta in nvds_obj_meta_list(frame_meta):

            quality_score = (
                obj_meta.confidence
                * min(obj_meta.rect_params.width * obj_meta.rect_params.height, 1.0)
            )

            # Push crop task to bounded async queue (non-blocking)
            try:
                crop_queue.put_nowait(CropTask(
                    surface_ref  = get_surface_ref(info, frame_meta),
                    obj_meta     = copy_obj_meta(obj_meta),
                    frame_meta   = copy_frame_meta(frame_meta),
                    quality_score= quality_score,
                ))
            except asyncio.QueueFull:
                metrics.increment("crop_queue.dropped")

            # Attach NvDsEventMsgMeta for nvmsgconv (metadata only, no pixels)
            attach_event_msg_meta(batch_meta, frame_meta, obj_meta)

            # Emit track lifecycle events
            if is_new_track(obj_meta.object_id):
                track_queue.put_nowait(TrackEvent(
                    track_id   = obj_meta.object_id,
                    event_type = "NEW",
                    stream_id  = frame_meta.source_id,
                    ts         = frame_meta.ntp_timestamp,
                ))
            elif is_lost_track(obj_meta.object_id):
                track_queue.put_nowait(TrackEvent(
                    track_id   = obj_meta.object_id,
                    event_type = "LOST",
                    stream_id  = frame_meta.source_id,
                    ts         = frame_meta.ntp_timestamp,
                ))

    return Gst.PadProbeReturn.OK

Async Crop Worker

async def crop_worker(crop_queue: asyncio.Queue, producer: AIOKafkaProducer):
    while True:
        task: CropTask = await crop_queue.get()

        try:
            # GPU → CPU transfer + JPEG encode (only blocking work)
            jpeg_bytes = await asyncio.to_thread(
                extract_and_encode_crop,
                task.surface_ref,
                task.obj_meta,
            )

            s3_key = None
            if len(jpeg_bytes) > INLINE_CROP_THRESHOLD:
                s3_key = await upload_to_s3(jpeg_bytes, task)
                jpeg_bytes = None

            topic = "vss.lp.crops" if task.obj_meta.class_id == LP_CLASS else "vss.objects.crops"

            await producer.send(topic, value={
                "stream_id"    : task.frame_meta.source_id,
                "frame_num"    : task.frame_meta.frame_num,
                "ts"           : task.frame_meta.ntp_timestamp,
                "track_id"     : task.obj_meta.object_id,
                "class_id"     : task.obj_meta.class_id,
                "bbox"         : extract_bbox(task.obj_meta),
                "quality_score": task.quality_score,
                "crop_jpeg"    : b64encode(jpeg_bytes) if jpeg_bytes else None,
                "s3_key"       : s3_key,
                "pipeline_id"  : PIPELINE_INSTANCE_ID,
            })

        except Exception as e:
            log.error("crop_worker failed: %s", e)
            metrics.increment("crop_worker.errors")

        finally:
            crop_queue.task_done()

Track Aggregator Consumer (Top Frame / Top-K)

class TrackAggregator:
    def __init__(self, k: int = 5, max_buffer: int = 50):
        self.k = k
        self.max_buffer = max_buffer
        self.buffers: dict[str, list] = defaultdict(list)  # track_id → heap

    def on_crop(self, msg: dict):
        tid = msg["track_id"]
        heap = self.buffers[tid]
        heapq.heappush(heap, (msg["quality_score"], msg["crop_jpeg"] or msg["s3_key"], msg["ts"]))
        if len(heap) > self.max_buffer:
            heapq.heappop(heap)  # drop lowest-quality

    def on_track_lost(self, event: dict):
        tid = event["track_id"]
        crops = sorted(self.buffers.pop(tid, []), reverse=True)

        if not crops:
            return

        self.kafka_producer.send("vss.tracks.summary", {
            "track_id"   : tid,
            "stream_id"  : event["stream_id"],
            "duration_ms": event["duration_ms"],
            "top_1_crop" : crops[0],
            "top_k_crops": crops[:self.k],
            "first_seen" : event["first_seen"],
            "last_seen"  : event["ts"],
        })

Operational Notes

Probe budget: Target < 1 ms per frame in the probe. The probe does zero I/O and zero encoding — only metadata copy, quality score computation (arithmetic), and queue push.

Queue sizing: Set crop_queue maxsize to ~2× your peak frame-rate × worker latency product. Drop oldest on full rather than blocking.

JPEG encode offload: extract_and_encode_crop (GPU→CPU + turbojpeg encode) typically takes 2–5 ms. Run in a thread pool (asyncio.to_thread) so it never touches the pipeline thread.

Crop size policy: Inline base64 for crops < 10 KB (LP regions). Use S3/MinIO + key for person/vehicle crops to keep Kafka message sizes bounded.

Track ID stability: nvtracker resets IDs on pipeline restart. Always include pipeline_instance_id (UUID generated at startup) in every message so consumers can detect restarts and flush stale state.

Kafka producer config (librdkafka):

queue.buffering.max.messages=100000
queue.buffering.max.ms=5
batch.num.messages=500
compression.type=lz4
message.max.bytes=5242880   # 5 MB — covers base64 person crop

Consumer scaling: - ReID Consumer: scale by camera zone, not by camera count (gallery is per-zone) - LP OCR Consumer: stateless, scale horizontally by partition count - Track Aggregator: one instance per stream_id partition to avoid cross-instance state - Analytics Consumer: stateless, scale freely

Concept: Frame-Synchronized Tracking

Tracking is a recurrence relation — the output at frame N depends on state from frame N-1. It must live where frames are guaranteed to be consecutive and in order.

Between frame N and frame N+1, the tracker:

Frame 1:  [bbox A at x=100]
Frame 2:  [bbox A at x=110]  → "moved 10px right, same object, track_id=1"
Frame 3:  [bbox A at x=120]  → "moved 10px right again, velocity=10px/f"
Frame 4:  NO DETECTION       → Kalman predicts bbox at x=130, track still alive
Frame 5:  [bbox A at x=138]  → close enough to prediction, re-associated

The tracker uses: - Kalman filter — predicts where an object will be next frame based on velocity - IoU / appearance matching — associates new detections to existing tracks - State machine — TENTATIVE → CONFIRMED → LOST based on consecutive hit/miss counts

Why it cannot be done downstream

Problem Detail
Pixels are gone By the time the consumer sees the message, the GPU frame buffer has been freed
No ordering guarantee Kafka does not guarantee per-frame ordering under producer batching or partition rebalancing
Latency compounds Tracker needs frame N to compute frame N+1’s prediction — buffering and waiting defeats real-time tracking
Association is atomic Hungarian algorithm matching across N detections and M tracks must happen per frame, not distributed

In DeepStream, nvstreammux enforces that every plugin sees the complete batch for timestamp T before any plugin sees T+1. This guarantee is enforced by the GStreamer pipeline clock and does not exist downstream of Kafka.

Concept: Raw Pixel Access

Raw pixel access means reading the actual RGBA/NV12 pixel values of a frame directly from GPU memory while it is still live in the pipeline.

When DeepStream decodes a frame it lands in an NvBufSurface in GPU memory. Every plugin — nvinfer, nvtracker, your probe — runs while that buffer is still allocated. Once the frame exits the pipeline the memory is freed. A Kafka message only carries numbers:

{ "bbox": [100, 200, 50, 80], "track_id": 42, "class": "person" }

The pixels are gone and cannot be reconstructed downstream.

Why DeepStream is ideal

The entire decode → infer → track chain is zero-copy on GPU. Your probe fires while the frame has already been used for inference — pixel access costs nothing extra.

Camera → decode → GPU memory
                     │
              ┌──────┴────────┐
              │  nvinfer      │  runs on GPU, no copy
              │  nvtracker    │  runs on GPU, no copy
              │  your probe   │  ← frame still in GPU memory here
              └───────────────┘
                     │
              frame released, memory freed

Concrete example — extracting a person crop

def unified_probe(pad, info, u_data):
    gst_buffer = info.get_buffer()
    batch_meta = pyds.gst_buffer_get_nvds_batch_meta(gst_buffer)

    for frame_meta in nvds_frame_meta_list(batch_meta):

        # Maps GPU frame buffer into a CPU-readable numpy array
        # Shape: (height, width, 4)  format: RGBA
        # This is the actual frame — every pixel of the camera feed at this moment
        frame_array = pyds.get_nvds_buf_surface(hash(gst_buffer), frame_meta.batch_id)

        for obj_meta in nvds_obj_meta_list(frame_meta):

            l = int(obj_meta.rect_params.left)
            t = int(obj_meta.rect_params.top)
            w = int(obj_meta.rect_params.width)
            h = int(obj_meta.rect_params.height)

            # Numpy slice into the live GPU-mapped buffer — no copy
            person_pixels = frame_array[t : t+h, l : l+w]

            # First and only copy — pixels → JPEG bytes → Kafka
            _, jpeg = cv2.imencode('.jpg', person_pixels, [cv2.IMWRITE_JPEG_QUALITY, 85])
            crop_queue.put_nowait(CropTask(jpeg=jpeg.tobytes(), track_id=obj_meta.object_id))

    return Gst.PadProbeReturn.OK

Downstream has no alternative

To get crops in a Kafka consumer without in-pipeline extraction you would need to send full frames (~500 KB JPEG at 30fps per camera = ~450 MB/s per camera). Extracting crops in the probe sends only the person region (5–20 KB).

Concept: Drawbacks of Multiple Probes

Every probe is a synchronous callback on the GStreamer streaming thread. The pipeline cannot advance to the next buffer until every probe returns.

1 — Every probe blocks the pipeline thread

Buffer arrives
    ├── Probe 1 fires  (5ms)   ← pipeline stopped
    ├── Probe 2 fires  (5ms)   ← pipeline stopped
    └── Probe 3 fires  (5ms)   ← pipeline stopped
                                  total stall: 15ms out of 33ms budget

2 — Metadata list traversed multiple times

Each probe that iterates the batch pays the full linked-list traversal cost again. With 8 cameras × 20 objects = 160 objects per batch, three probes do 480 object iterations instead of 160.

3 — Shared state requires locks

If probe 1 computes a quality score and probe 2 needs it, the data must be shared via a dict with a lock. Probes on different pads can fire from different threads — a real race condition without explicit synchronisation.

4 — Execution order is implicit

GStreamer evaluates probes in the order add_probe() was called. If a crop probe is added before an annotation probe, crops are extracted without annotations — silent wrong behaviour with no error.

5 — NvBufSurface mapping cost multiplies

Each probe that calls pyds.get_nvds_buf_surface() pays an independent GPU→CPU map/unmap. Each map is a CPU–GPU synchronisation point. One probe maps once and extracts everything; multiple probes each pay this independently.

Drawback Impact
Pipeline stall Latency multiplies linearly with probe count
Metadata traversal CPU cost multiplies with probe count
Shared state Race conditions on multi-pad pipelines
Implicit order Silent wrong behaviour when order matters
Surface remap Unnecessary GPU↔︎CPU sync points
Harder to debug No clear owner when things go wrong

Rule: one probe, one traversal, collect everything into local variables, dispatch to async workers.

Concept: Why ReID Belongs Downstream

ReID answers: “Is the person in camera B the same person I saw in camera A 2 minutes ago?” It works by extracting a 512-dim embedding from a crop and comparing it against a gallery of known embeddings.

It is not frame-synchronous

Tracking needs:  frame 1 → frame 2 → frame 3   (ordered, no gaps)
ReID needs:      crop @ frame 847 → compare against gallery
                 (result is identical whether run 1ms or 1s later)

The crop is already available downstream

The probe extracts the JPEG crop and publishes it to vss.objects.crops. The ReID consumer has the actual pixels without touching the pipeline at all.

Gallery management is complex stateful logic

The gallery grows, persists, ages out, and spans cameras. It requires FAISS or Redis vector search, per-zone partitioning, and entry expiry. None of this belongs in a probe called 30 times per second.

GPU contention with the pipeline

Pipeline GPU budget per frame at 30fps = 33ms
Primary detector:   ~8ms
nvtracker:          ~5ms
Secondary detector: ~4ms
─────────────────────────
Remaining:          ~16ms

ReID per crop (OSNet): ~6ms × 20 objects = 120ms  ← blows the budget

Downstream ReID runs on its own GPU or schedule without affecting pipeline FPS.

Cross-camera matching requires multi-stream context

A probe only has context for its own pipeline. Cross-camera matching requires seeing crops from all cameras simultaneously — which is only possible in a consumer subscribed to all streams on one topic.

The one exception

nvtracker can use a ReID model internally (via re-id-model-path in config) to re-associate lost tracks within a single camera. This is frame-synchronous and belongs in the pipeline. It is separate from cross-camera or long-term person identification, which always belongs downstream.

Concept: Why LP OCR Belongs Downstream

The secondary nvinfer detects the LP bbox. The probe extracts the LP crop and sends it to vss.lp.crops. OCR is: pixels in → plate string out — it has no dependency on the live frame or consecutive frames.

It is not frame-synchronous

OCR on a crop produces the same plate string whether run 1ms or 1 second after the frame. Unlike tracking, there is no correctness dependency on timing.

You should not OCR every frame

A car stopped for 10 seconds at 30fps = 300 LP crops. Running OCR on all 300 is wasteful. The correct approach:

300 crops for track_id=42
    │
    ├── Track Aggregator picks top-3 sharpest crops (by quality_score)
    └── OCR Consumer runs on 3 crops → majority vote → confident result

This selection requires buffering per track and flushing on TRACK_LOST — stateful logic that cannot run in a per-frame probe.

OCR engines are not pipeline-native

Engine Runtime Per-crop latency
PaddleOCR Python / ONNX 20–80ms
EasyOCR Python / PyTorch 50–150ms
Tesseract C++ / CPU 30–100ms
TRT-OCR TensorRT 5–15ms

Even TRT-OCR at 15ms running on every LP detection in an 8-camera 30fps pipeline: 8 × 30 × 15ms = 3600ms/s of OCR work against a 33ms/frame budget — pipeline completely stalled.

def normalize_plate(raw: str) -> str:
    cleaned = raw.upper().replace(" ", "").replace("-", "")
    return cleaned.replace("O", "0").replace("I", "1").replace("S", "5")

Concept: Frame-Level vs Track-Level Storage

Where Track-Level Is Sufficient

For identity and evidence-based use cases, vss.tracks.summary is all you need to persist:

Use Case Track-Level Enough?
Who was at the site Yes — first_seen, last_seen, top_1_crop
Best photo of each person Yes — top_1_crop
Total dwell time Yes — duration_ms
LP of a vehicle Yes — emitted on TRACK_LOST
ReID / cross-camera match Yes — crops already present

For these, vss.detections is a transient stream consumed in real-time and never stored.

Where Frame-Level Cannot Be Replaced

Zone crossing / tripwire alerts

Person trajectory:
  frame 100: bbox at x=300  (left of line)
  frame 101: bbox at x=480  (left of line)
  frame 102: bbox at x=520  ← crossed line at x=500
  frame 103: bbox at x=600  (right of line)

Track summary only gives first_seen and last_seen. It cannot tell you when a person crossed a boundary or which zones they passed through. Every frame’s bbox is required to detect the crossing moment.

Zone-specific dwell time

Track duration: 120 seconds   ← track summary knows this

Zone A (entrance):   45 seconds  ┐
Zone B (shelf):      60 seconds  ├── only frame-level positions reveal this
Zone C (checkout):   15 seconds  ┘

Trajectory and path analysis — heat maps, flow direction, and common paths all require aggregating bbox positions across frames, not just start and end points.

Speed estimation — speed = delta_bbox / delta_time needs at least two consecutive frame positions. Track summary has none.

Crowd density at a point in time — “How many people were in Zone B at 10:32:15?” Track summary only says the track existed between first_seen and last_seen, not where it was at a specific moment.

The Right Storage Split

vss.detections is a real-time stream for alert consumers — it is never written to a database. Only vss.tracks.summary hits long-term storage.

vss.detections  ──► Real-time consumers only        (NOT persisted)
                    • tripwire / zone crossing
                    • crowd density threshold
                    • speed alerts

vss.tracks.summary ──► Database                     (persisted)
                        • identity store
                        • dwell time
                        • crops / evidence

If historical trajectory is needed, the alert consumer writes a downsampled path on TRACK_LOST — not the raw 30fps stream:

# Written to DB on TRACK_LOST — 1 sample per second, not per frame
{
    "track_id": 42,
    "trajectory": [
        {"ts": 1000, "x": 300, "y": 200, "zone": "A"},
        {"ts": 2000, "x": 450, "y": 210, "zone": "B"},
        {"ts": 3000, "x": 600, "y": 215, "zone": "C"},
    ]
}

This reduces storage by 30× (1 sample/s vs 30 samples/s) while preserving enough resolution for zone and path queries.

Concept: Threading Model for Probe → Async Workers

The Core Problem

GStreamer runs the probe on its streaming thread — a synchronous C-level callback. Python’s asyncio is a single-threaded event loop. These two worlds cannot talk directly and need an explicit bridge.

GStreamer Streaming Thread  (synchronous, C-level)
        │
        │  ← probe fires here, must return in < 1ms
        │
        ?  ← how does sync talk to async?
        │
asyncio Event Loop Thread   (async, I/O-efficient)
ThreadPoolExecutor Threads  (CPU-bound work: JPEG encode, GPU→CPU)

The Bridge: thread-safe queue between sync and async

asyncio.Queue is not thread-safe — you cannot call put_nowait() on it from the GStreamer thread. Use queue.Queue (stdlib, thread-safe) as the bridge.

Concept: Why JPEG Encoding Runs in a Thread Pool

What JPEG Encoding Actually Does

JPEG is a pipeline of five sequential steps:

Raw pixels (RGB)
    │
    ▼
1. Color space conversion    RGB → YCbCr
    │
    ▼
2. Chroma downsampling       discard half the color resolution (human eyes don't notice)
    │
    ▼
3. Block splitting           cut image into 8×8 pixel blocks
    │
    ▼
4. DCT per block             Discrete Cosine Transform — math transform on each 8×8 block
    │
    ▼
5. Huffman encoding          variable-length entropy coding — inherently sequential
    │
    ▼
JPEG bytes

Steps 1–4 are parallelisable. Step 5 is not — and step 5 is the bottleneck.

Why Huffman Encoding Is Sequential

Huffman coding builds a variable-length bit stream where each output bit’s position depends on all the bits before it:

Block 1 output: 10 bits
Block 2 output: starts at bit 11  ← must know block 1 finished at bit 10
Block 3 output: starts at bit 23  ← must know block 2 finished at bit 22

You cannot start encoding block 3 until block 2 is done. This is a true sequential dependency — it cannot be parallelised. This is why JPEG encoding is CPU-bound, not GPU-bound.

Why GPU Is Not Ideal for Small Crops

Your crops are small — a person bounding box is typically 80×200 px = 200 eight-by-eight blocks. For that size, GPU overhead dominates:

GPU path (nvjpeg):
  CPU → GPU transfer of crop:       0.5 ms   ← crop is already on CPU after surface map
  GPU kernel launch overhead:       0.2 ms
  GPU DCT + quantise (200 blocks):  0.1 ms
  GPU Huffman (still sequential):   0.3 ms
  GPU → CPU transfer of JPEG:       0.3 ms
  Total:                            1.4 ms

CPU path (turbojpeg with SIMD):
  DCT + quantise:                   0.1 ms
  Huffman:                          0.2 ms
  Total:                            0.3 ms

GPU is slower for small crops because transfer overhead dominates. nvjpeg becomes worthwhile for full frames (1920×1080 = 32,400 blocks) or batch-encoding hundreds of crops simultaneously.

Why turbojpeg Is Fast on CPU

turbojpeg wraps libjpeg-turbo which uses SIMD instructions — SSE2, AVX2 on x86, NEON on ARM. These process multiple 8×8 blocks simultaneously within a single CPU core:

AVX2: 256-bit registers
  → 4 × float64 DCT values in one instruction
  → effectively 4× throughput on DCT vs scalar code

Why It Belongs in the Thread Pool, Not the Event Loop

JPEG encoding is pure CPU computation — there is no I/O to await. If it ran directly in the asyncio event loop coroutine, it would block the event loop thread for ~0.3 ms per crop, preventing all other coroutines (Kafka produce, S3 upload) from running during that time.

# WRONG — blocks the event loop
async def crop_worker():
    task = await queue.get()
    jpeg = turbojpeg.encode(pixels)   # 0.3ms of CPU — event loop frozen here
    await kafka_producer.send(...)    # nothing else can run until encode finishes

# CORRECT — offloads CPU work to thread pool, event loop stays free
async def crop_worker():
    task = await queue.get()
    jpeg = await loop.run_in_executor(executor, turbojpeg.encode, pixels)
    #      ↑ event loop suspends this coroutine, runs others while thread encodes
    await kafka_producer.send(...)

run_in_executor runs turbojpeg.encode in a ThreadPoolExecutor thread. turbojpeg is a C extension and releases the GIL during encoding — giving true CPU parallelism across multiple crops simultaneously.

The Full Pixel Journey

NvBufSurface (GPU memory)
    │
    │  pyds.get_nvds_buf_surface()

    ▼
numpy array [H, W, 4]             ← pointing into mapped GPU memory

Implementation Details

1. Surface Reference Lifecycle

pyds.get_nvds_buf_surface() maps GPU memory that is only valid while the probe is executing. Once the probe returns, GStreamer recycles that buffer. Accessing a surface reference after the probe returns produces corrupted pixels or a segfault.

Rule: copy crop pixels inside the probe. Push raw pixels to the queue. JPEG encode in the worker.

# IN the probe (GStreamer thread)
frame_array = pyds.get_nvds_buf_surface(hash(gst_buffer), frame_meta.batch_id)

l, t = int(obj_meta.rect_params.left), int(obj_meta.rect_params.top)
w, h = int(obj_meta.rect_params.width), int(obj_meta.rect_params.height)

# .copy() is mandatory — detaches from mapped GPU memory
crop_pixels = frame_array[t:t+h, l:l+w].copy()

crop_queue.put_nowait(CropTask(pixels=crop_pixels, ...))
# probe returns — surface safely recycled by GStreamer

# IN the worker thread (safe — plain numpy array in CPU memory)
jpeg = turbojpeg.encode(task.pixels)

The .copy() is a small numpy memcpy of just the crop region, not the full frame. At 80×200 RGBA = 64 KB — fast enough to do inside the probe.

2. TRACK_NEW / TRACK_LOST Detection

Maintain a per-source set of active track IDs and diff on every frame. Add a grace period to avoid false LOST events from temporary occlusion — nvtracker can drop a track for 1–2 frames and re-emit it with the same track_id.

active_tracks     : dict[int, set[int]] = defaultdict(set)   # source_id → track_ids
track_absent_count: dict[int, int]      = defaultdict(int)   # track_id  → absent frames

LOST_GRACE_FRAMES = 5   # track must be absent for 5 consecutive frames before LOST

def detect_track_events(frame_meta) -> tuple[set, set]:
    source_id = frame_meta.source_id
    current   = {o.object_id for o in nvds_obj_meta_list(frame_meta)}
    previous  = active_tracks[source_id]

    new_tracks = current - previous

    confirmed_lost = set()
    for tid in (previous - current):
        track_absent_count[tid] += 1
        if track_absent_count[tid] >= LOST_GRACE_FRAMES:
            confirmed_lost.add(tid)

    for tid in current:
        track_absent_count.pop(tid, None)   # reset if track reappears

    active_tracks[source_id] = current
    return new_tracks, confirmed_lost

4. obj_meta Copy Strategy

obj_meta is a pyds wrapper around a C pointer that becomes invalid after the probe returns. Do not hold a reference to the raw obj_meta object. Copy only the scalar fields you need into a plain Python dataclass.

from dataclasses import dataclass

@dataclass
class ObjMetaCopy:
    object_id : int
    class_id  : int
    confidence: float
    left      : float
    top       : float
    width     : float
    height    : float

def copy_obj_meta(obj_meta) -> ObjMetaCopy:
    return ObjMetaCopy(
        object_id  = obj_meta.object_id,
        class_id   = obj_meta.class_id,
        confidence = obj_meta.confidence,
        left       = obj_meta.rect_params.left,
        top        = obj_meta.rect_params.top,
        width      = obj_meta.rect_params.width,
        height     = obj_meta.rect_params.height,
    )

Do not copy the full obj_meta struct — it contains C pointers that are invalid outside the probe.

5. Graceful Shutdown

Without explicit shutdown handling, tracks that are active when the process receives SIGTERM never produce a vss.tracks.summary message. Their crops are orphaned in Redis.

import signal

async def shutdown(loop):
    # 1. Stop the pipeline from producing new frames
    pipeline.set_state(Gst.State.PAUSED)

    # 2. Drain the crop queue — wait for all enqueued crops to be encoded and sent
    await asyncio.wait_for(
        asyncio.get_event_loop().run_in_executor(None, crop_queue.join),
        timeout=10.0
    )

    # 3. Emit TRACK_LOST for all still-active tracks
    for source_id, track_ids in active_tracks.items():
        for tid in track_ids:
            track_queue.put_nowait(TrackEvent(
                track_id   = tid,
                event_type = "LOST",
                source_id  = source_id,
                ts         = current_timestamp(),
            ))

    # 4. Drain track event queue
    await asyncio.wait_for(
        asyncio.get_event_loop().run_in_executor(None, track_queue.join),
        timeout=5.0
    )

    # 5. Flush Kafka — ensure all buffered messages are delivered
    await kafka_producer.flush()

    # 6. Stop pipeline fully
    pipeline.set_state(Gst.State.NULL)
    loop.stop()

# Register handler
signal.signal(signal.SIGTERM, lambda s, f: asyncio.create_task(shutdown(loop)))
signal.signal(signal.SIGINT,  lambda s, f: asyncio.create_task(shutdown(loop)))

6. Quality Score Formula

The placeholder confidence × bbox_area does not account for blur or occlusion. A production quality score for ReID and top-frame selection:

def compute_quality(obj_meta, frame_array: np.ndarray) -> float:
    l = int(obj_meta.rect_params.left)
    t = int(obj_meta.rect_params.top)
    w = int(obj_meta.rect_params.width)
    h = int(obj_meta.rect_params.height)

    # Size: larger crops have more detail
    TARGET_W, TARGET_H = 128, 256
    size_score = min((w * h) / (TARGET_W * TARGET_H), 1.0)

    # Aspect ratio: penalise heavily occluded or oddly-cropped boxes
    aspect = h / max(w, 1)
    aspect_score = 1.0 if 1.5 < aspect < 4.0 else 0.5

    # Blur: Laplacian variance — sharp images have high variance
    # Compute here while frame_array is still mapped (before probe returns)
    crop = frame_array[t:t+h, l:l+w]
    gray = cv2.cvtColor(crop, cv2.COLOR_RGBA2GRAY)
    blur_score = min(cv2.Laplacian(gray, cv2.CV_64F).var() / 500.0, 1.0)

    return obj_meta.confidence * size_score * aspect_score * blur_score

Blur score must be computed in the probe while frame_array is mapped — not in the worker thread where the surface is no longer valid.
