§ 01 Premise

The tracking problem

A detector tells you what is in each frame. A tracker tells you which one is which across frames. NvDCF is the bridge.

The primary inference engine (PGIE) in a DeepStream pipeline produces bounding boxes per frame. Those boxes have no memory: the "person" detected in frame N and the "person" detected in frame N+1 are anonymous to each other. The tracker's job is to look at both and decide whether they are the same person (carry the same object_id) or different people.

This sounds easy when one object walks through an empty scene. It becomes hard when:

  • Two objects pass close to each other (their bounding boxes overlap).
  • The detector misses a frame (one of the two objects briefly disappears from the input).
  • An object is occluded for half a second (a person walks behind a pillar).
  • Two objects look similar (two delivery drivers in the same uniform).

NvDCF combines four independent signals to handle these cases. Each signal compensates for the others' weaknesses.

§ 02 Setup

Two frames, one identity

Before we look at any single component, here is the concrete situation the tracker faces every 33 milliseconds. Two consecutive frames from a single stream. Two people, ID 1 and ID 2, walking towards each other.

In an easy frame like this one — the two people are spatially separated and moving in opposite directions — almost any heuristic works. IoU alone would give the right answer. The interesting question is what happens when the situation is not easy. The rest of this document walks through each signal the tracker uses, with the same two-person scene as the running example, slowly making the scene harder until you need all four signals working together.

§ 03 Signal 01 · Visual

Discriminative Correlation Filter

Per target, learn a small image template that lights up when it sees that target and goes quiet on everything else. Slide it across the next frame and find the brightest spot.

The intuition

Each tracked target carries a learned filter: a small patch of image-shaped weights, the same size as the target's appearance template. Cross-correlating this filter with a region of the next frame produces a response map — a 2D heatmap where bright peaks mean "the filter matches strongly here". The peak is the new bounding box center; the peak's sharpness is the tracker's confidence.

"Discriminative" means the filter was trained — across many past frames — to produce a sharp peak on the target and a flat low response on the background around it. That's what makes it robust to small appearance changes: it learned what to ignore.

The math, in two lines

The filter h is learned by minimising, over a window of recent frames, the squared error between the actual response map and an ideal Gaussian-shaped target response (sharp peak at the target center, zero elsewhere):

// minimise over h:
  Σ_i || h ⋆ f_i  −  g_target ||²  +  λ ||h||²

// closed-form solution in the frequency domain:
  H = (F̂* · Ĝ) / (F̂* · F̂ + λ)

where F̂ is the FFT of the feature patch, Ĝ is the FFT of the ideal Gaussian response, and λ is a regularisation constant. Each frame, the filter is updated with a learning rate:

  H_t = (1 − η) · H_{t-1}  +  η · H_new

where η = filterLr. Low η means a stable filter that ignores transient appearance changes; high η means the filter adapts fast but can drift onto an occluder.

Features, not raw pixels

NvDCF does not correlate over raw RGB. It extracts features from each crop — historically HOG and colour-names in the perf config, optionally deep CNN features in the accuracy config — and learns the filter in feature space. This is what gives the tracker robustness to illumination changes and minor appearance shifts that raw pixel correlation would fail on.

Peak-to-Sidelobe Ratio (PSR)

The peak value alone isn't useful — a bad response that happens to be 0.4 doesn't mean less than a sharp response that peaks at 0.3. NvDCF uses PSR: how much the peak exceeds the mean of the rest of the response map, normalised by the sidelobe standard deviation. A confident track has PSR > 8; a target being lost falls below 5.

  PSR = (peak − μ_sidelobe) / σ_sidelobe

  if PSR < minTrackerConfidence:
      demote target to shadow mode

Why this fails alone

The DCF filter learns whatever sits in its search region. If a similar-looking target enters that region, the filter happily learns the new appearance and the track silently switches. This is why NvDCF combines DCF with motion (which can't be fooled by appearance) and ReID (which is global, not per-target-learned).

§ 04 Signal 02 · Motion

Kalman motion model

Track each target's position and velocity, and keep a running estimate of how uncertain that prediction is.

The DCF can tell where in the search region a target is, but it can't tell you where the search region should be in the first place. That's the motion model's job. NvDCF uses a Kalman filter per target with state [x, y, vx, vy] (some configurations also include scale).

Predict, then update

Every frame the Kalman filter does two operations:

  • Predict. Apply the motion model: new position = old position + velocity·Δt. The position estimate gets pushed forward, and the uncertainty ellipse grows by the process noise Q.
  • Update. Once a measurement arrives (the matched detection or DCF peak), correct the prediction towards it, weighted by how much we trust each. The uncertainty ellipse shrinks.

Why uncertainty matters

The Kalman covariance isn't an aesthetic detail — it directly controls the data association in the next step. When a target's uncertainty ellipse is small (recently observed with high confidence), the associator gives a very low cost to detections that fall inside it and a very high cost to detections outside it. When the ellipse is large (the target has been in shadow for several frames), the associator is more permissive about where the detection can be — which is what lets the tracker recover from short occlusions.

Mahalanobis distance, not Euclidean

The association cost from the motion side isn't (x − x_pred)² + (y − y_pred)². It's the Mahalanobis distance, which divides by the uncertainty:

  d_M² = (z − ẑ)ᵀ · S⁻¹ · (z − ẑ)

where z is the detection, ẑ is the prediction, and S is the innovation covariance. The same 30-pixel gap between prediction and detection is "small" when the target's been in shadow for 10 frames (ellipse is big) and "huge" when the target was just observed (ellipse is tight). This is what makes long occlusions still work without the tracker greedily snapping to any nearby detection.

Why this fails alone

Motion is blind to identity. Two people walking towards each other have crossing trajectories; at the moment of the cross, motion alone cannot tell which is which. Motion also breaks when an object suddenly stops or reverses (its predicted position is now far from reality). DCF and ReID handle exactly these failure modes.

§ 05 Signal 03 · Identity

ReID embeddings

A small CNN turns each cropped object into a fingerprint vector. Two vectors are close if they came from the same physical object; far if they came from different ones.

Where DCF learns a per-target filter online and Kalman tracks per-target motion, ReID provides a global appearance signal: a fixed model, trained offline on many identities, that knows how to map any crop of "a person" into a vector such that the same physical person produces nearby vectors across views, lighting, and time.

The bundled NvDCF ReID model in DeepStream's accuracy config is a ResNet-50 trained on Market-1501. It outputs a 256-dimensional unit-normalised embedding per crop.

How ReID feeds into association

The cosine similarity gets converted to a cost (lower is better) by negating and shifting: c_reid = 1 − cos(e_a, e_b). This is one of three terms in the association cost matrix, alongside IoU and Mahalanobis distance.

Embedding gallery, not snapshot

NvDCF doesn't compare against a single embedding per track — it maintains a small gallery (last N embeddings, exponential moving average, or both, depending on config). This makes the comparison robust to a single bad crop (motion blur, half-occluded frame) without letting outdated embeddings dominate.

Cost vs benefit

ReID is the most expensive component. Each per-target inference costs a CNN forward pass (~2-5 ms on Orin Nano for ResNet-50 at 128×256). With 30 targets per frame, that's 60-150 ms — far beyond a real-time budget. Two knobs make this affordable:

  • reidExtractionInterval: extract embeddings every N frames per target (N=3-5 typically). Embeddings change slowly; you don't need them every frame.
  • batchSize: batch all targets' crops into a single inference call. The CNN's forward pass is largely fixed cost per batch.

Why this fails alone

ReID is trained on a fixed domain. Generic person-ReID models give noisy embeddings on vehicles, animals, or factory parts. Two people in the same uniform also produce embeddings close enough to confuse the cost matrix. ReID needs IoU and motion to anchor it in the physical scene.

§ 06 Signal 04 · Assignment

Hungarian assignment

Three signals give you a cost matrix. The Hungarian algorithm gives you the globally optimal one-to-one assignment from detections to tracks in polynomial time.

For each (track, detection) pair, the tracker now has three cost contributions. NvDCF combines them as a weighted sum:

  C[i, j] = w_iou · (1 − IoU(track_i, det_j))
          + w_mhd · d_M(track_i, det_j)
          + w_reid · (1 − cos(e_i, e_j))

where w_iou + w_mhd + w_reid = 1 and the individual weights live in the tracker YAML. This builds a cost matrix:

Why Hungarian, not greedy

Greedy assignment — pick the cheapest cell, remove its row and column, repeat — works when the cheapest pair is unambiguous. It fails when two tracks both have their lowest cost on the same detection: greedy commits to the first one and forces the second to a worse match. Hungarian considers the whole matrix simultaneously and reshuffles to find the global optimum.

Two-stage matching

Recent NvDCF (DeepStream 6.3+) borrows from ByteTrack: it runs the assignment twice.

  • Stage 1: High-confidence detections matched against active tracks. This handles 90% of the work cleanly.
  • Stage 2: Low-confidence detections matched against the still-unmatched tracks from Stage 1. This recovers partially-occluded objects whose detections were filtered out earlier — they'd have been wasted otherwise.

The threshold dividing the two stages is tentativeDetectorConfidence.

Cost gating

Before running Hungarian, the tracker discards cells whose cost exceeds a threshold (maxMatchingCost). This prevents the optimizer from making a "least bad" assignment between two unrelated objects just because no better option exists. Gated cells become "unmatched" — the track moves to shadow, the detection becomes a new tentative track.

§ 07 Lifecycle

Target lifecycle

A track passes through four states. The transitions between them are the difference between a stable tracker and one that flickers identities every few frames.

Shadow mode and past-frame meta

When an active track is unmatched for a frame, it does not die. It transitions to shadow mode, where the Kalman filter keeps predicting position frame by frame but no output is emitted. If a detection re-associates with the shadow track within maxShadowTrackingAge frames, the track is reactivated — and the frames spent in shadow are emitted retroactively as NvDsPastFrameObjBatch user-meta on the batch.

This retroactive emission is the reason consumers see "the same person" across a brief occlusion: the ID never changed, only the visibility did.

ID stability vs ID switches

Two failure modes pull maxShadowTrackingAge in opposite directions:

  • Too low: a person walks behind a 2-second pillar, the shadow times out, and they get a new ID when they emerge. This is an ID switch.
  • Too high: target A walks behind a pillar, target B walks out from behind it. The shadow track for A re-matches with B's detection. A's identity attaches to B's body. This is also an ID switch, but worse — it's silently wrong rather than visibly missed.

The right value depends on scene geometry (how long are typical occlusions?) and crowd density (how likely is a different object to appear in the same place?).

Tuning workflow

  1. Start with the bundled config_tracker_NvDCF_accuracy.yml unchanged. Run on a representative clip.
  2. Measure ID switches per object-minute on your scene. If under 1, stop — you're done.
  3. If switches are high, first sweep maxShadowTrackingAge (try 15, 30, 60, 90). Most scenes have a clear optimum.
  4. If switches persist on fast motion, raise processNoiseVar4Loc (Kalman accepts faster movement).
  5. If switches persist on visually similar objects, raise matchingScoreWeight4Reid at the cost of matchingScoreWeight4Iou.
  6. If latency is too high, switch to perf config and set reidExtractionInterval = 5 as a starting point.
